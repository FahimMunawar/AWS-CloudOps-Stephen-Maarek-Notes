# 8. CloudFront Logging, Reports, and Monitoring

## Overview

CloudFront provides **access logging** for full request-level audit trails, **built-in reports** for traffic analysis, and **CloudWatch metrics** for real-time monitoring — all without requiring third-party tools.

---

## Access Logs

CloudFront logs every request made to the distribution into a designated **S3 logging bucket**.

### Architecture

```
Users → CloudFront Edge Locations (global)
              │
              │  log data
              ▼
        S3 Logging Bucket  (separate from origin bucket)
              │
              ├── Athena (SQL queries on log files)
              ├── CloudWatch Logs (real-time)
              └── Kinesis Data Firehose (streaming)
```

### Key Points

| Setting | Detail |
|---------|--------|
| **Enable in** | Distribution → Settings → Edit → Standard logging → On |
| **Destination** | An S3 bucket (separate from your origin bucket) |
| **Log prefix** | Optional — use to separate logs from multiple distributions in one bucket |
| **Multiple distributions** | Can all log to the same S3 bucket using different prefixes |
| **Format** | Tab-delimited; each line = one request |

### Log Fields (Examples)

Each log entry includes: date, time, HTTP method, path, status code, user-agent, bytes transferred, edge location, and more.

```
date    time     method   path       status   user-agent    ...
2024-01 12:00:00  GET   /beach.jpg    403     Mozilla/5.0   ...
```

> **Note:** The logging S3 bucket should be **different** from the S3 origin bucket.

### Downstream Analysis

- **Amazon Athena** — run SQL queries against the log files in S3
- **CloudWatch Logs** — real-time log streaming
- **Kinesis Data Firehose** — real-time data processing pipelines

## CloudFront Built-In Reports

Available under: **CloudFront → Reports & Analytics**

> Reports are generated by CloudFront regardless of whether access logs are enabled in S3.

| Report | What It Shows |
|--------|--------------|
| **Cache Statistics** | Requests over time, cache hit/miss ratio, bytes transferred, HTTP status code breakdown (2xx/3xx/4xx/5xx) |
| **Popular Objects** | Most-requested objects with hit/miss counts |
| **Top Referrers** | Which external websites are sending traffic to your distribution |
| **Usage** | Data transfer volume by protocol (HTTP vs HTTPS), CloudFront-to-viewer and CloudFront-to-origin |
| **Viewers** | Device type (desktop/mobile/bot), browser, OS, geographic location |

**Granularity:** Daily or hourly views available.

## HTTP Status Code Reference (Troubleshooting)

CloudFront **caches error responses** from the origin just like successful responses.

| Status Range | Meaning |
|-------------|---------|
| **4xx** | Client-side issue — user doesn't have access (403) or object doesn't exist (404) |
| **5xx** | Server-side issue — gateway errors, origin unavailable |

> If a 4xx or 5xx response is cached, users will continue to see the error until the cache TTL expires or an invalidation is run.

## CloudWatch Monitoring

**Console:** CloudFront → select distribution → **Monitoring**

| Metric | What It Measures |
|--------|-----------------|
| **Requests** | Total request count over time |
| **Data transfer** | Bytes transferred to viewers |
| **Error rate (4xx)** | Percentage of requests returning 4xx |
| **Error rate (5xx)** | Percentage of requests returning 5xx |
| **Origin latency** | Response time from origin |
| **Cache hit rate** | Percentage of requests served from cache |

---

## Best Practices

✓ **Use a separate S3 bucket for logs** — never log into your origin bucket  
✓ **Use prefixes when logging multiple distributions** to the same bucket  
✓ **Use Athena to query access logs** for ad-hoc analysis without a dedicated logging service  
✓ **Monitor cache hit rate in CloudWatch** — a healthy distribution typically achieves 70–80%+  
✓ **Set up CloudWatch alarms on 5xx error rate** — spikes indicate origin availability issues  
✓ **Enable access logging for compliance and debugging** — built-in reports don't provide request-level detail

## SysOps Exam Focus

**Q1: "Where does CloudFront store access logs when standard logging is enabled?"**
- A) CloudWatch Logs automatically
- B) An S3 bucket that you specify (should be separate from the origin bucket)
- C) The CloudFront distribution itself
- D) AWS X-Ray traces
- **Answer: B** — You specify an S3 logging bucket; it should be separate from the origin bucket

**Q2: "Do you need to enable S3 access logging to use CloudFront's built-in reports (cache statistics, viewers, etc.)?"**
- A) Yes — reports require logs to be written to S3 first
- B) No — CloudFront generates these reports independently of the S3 logging setting
- C) Yes — but only for the cache statistics report
- D) No — but only if you use Kinesis Data Firehose
- **Answer: B** — Built-in reports are always available regardless of whether S3 logging is enabled

**Q3: "A CloudFront distribution is returning 403 errors to users even though the file exists in S3. What is a likely cause?"**
- A) CloudFront is not configured with an origin
- B) OAC is not set up and the S3 bucket is private — CloudFront cannot access the object
- C) The cache TTL has not expired
- D) The edge location is too far from the origin
- **Answer: B** — A 403 from CloudFront to S3 typically means missing OAC/bucket policy, preventing CloudFront from reading the private bucket

**Q4: "You want to analyze which countries are generating the most traffic to your CloudFront distribution. Where do you find this?"**
- A) CloudFront → Security → Geo restrictions
- B) CloudFront → Reports & Analytics → Viewers → Location
- C) CloudWatch → CloudFront metrics → Geographic
- D) AWS Cost Explorer
- **Answer: B** — The Viewers report includes geographic location data showing which countries generate traffic

**Q5: "How can you run SQL queries against CloudFront access logs?"**
- A) Use the CloudFront Reports console
- B) Use Amazon Athena with the S3 logging bucket as the data source
- C) Use AWS Glue to query CloudFront logs directly
- D) CloudFront logs cannot be queried — only viewed in the console
- **Answer: B** — Athena queries S3 directly; point it at the logging bucket to run SQL against access log files

## Quick Reference

```
Access Logs:
  Enable: Distribution → Settings → Standard logging → On → S3 bucket
  Destination: separate S3 bucket (not origin bucket)
  Use prefixes for multiple distributions in one bucket
  Analysis: Athena (SQL) | CloudWatch Logs | Kinesis Firehose

Built-in Reports (no S3 logging required):
  Cache Statistics  →  hit/miss ratio, status codes
  Popular Objects   →  most-requested content
  Top Referrers     →  traffic sources
  Usage             →  data transfer by protocol
  Viewers           →  device, browser, OS, country

CloudWatch Metrics:
  Requests, Data transfer, 4xx rate, 5xx rate, Origin latency, Cache hit rate

Error caching:
  CloudFront caches 4xx/5xx responses — invalidate to clear error cache
```
